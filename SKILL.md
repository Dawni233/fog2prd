# Fog2PRD: From Fuzzy Requirements to Ship-Ready Specs

Turn raw, ambiguous product requirements into structured PRD documents, interactive prototypes, and developer-ready handover specs — in one guided pipeline.

## 触发条件

当用户说以下关键词时触发：
- "设计需求"、"生成PRD"、"需求分析"
- "需求文档"、"原型设计"
- 提供需求描述并请求文档/原型输出

---

## 工作流程概览

```
用户需求 → [阶段1: 需求理解与澄清] → [阶段2: 原型设计与迭代] → [阶段3: 需求规范生成] → [阶段4: 需求传递单生成] → 交付
              ↓                           ↓                        ↓                      ↓
         需求理解报告                确认版原型               需求规范(签字版)        需求传递单(开发版)
```

### 核心设计理念

| 理念 | 说明 |
|------|------|
| **原型优先** | 先出原型让用户确认，再生成正式文档 |
| **迭代确认** | 原型阶段支持多轮迭代，直到用户满意 |
| **双文档输出** | 用户签字版（正式存档） + 开发传递版（高效交付） |
| **轻量传递** | 开发版聚焦关键信息，5分钟可理解 |

### 阶段说明

| 阶段 | 目标 | 输出 | 用户参与 |
|------|------|------|----------|
| 阶段1 | 理解需求，澄清遗漏 | 需求理解报告 | 回答澄清问题 |
| 阶段2 | 原型确认，迭代修改 | HTML原型 + 业务流程图 | 体验原型，反馈修改 |
| 阶段3 | 正式文档，签字确认 | 需求规范文档 | 签字确认 |
| 阶段4 | 开发交付，高效传递 | 需求传递单 + 页面线框图 | 无（交付开发） |

> **效率优化**：页面线框图延迟到阶段4生成（低质量快速模式），避免在阶段2迭代时因生成图表耗时过长。

## 阶段1: 需求理解与澄清

### 1.1 输入类型

| 类型 | 说明 |
|------|------|
| 口头需求 | 用户口述的文字记录 |
| 高层级文档 | 已有的需求说明文档 |

### 1.2 处理步骤

1. **需求解析**
   - 提取用户角色
   - 识别功能点
   - 发现歧义点

2. **复杂度评估**

   评分维度：

   | 维度 | 简单(1分) | 中等(2-4分) | 复杂(5分) |
   |------|-----------|-------------|-----------|
   | 功能点数量 | ≤3个 | 4-8个 | >8个 |
   | 涉及角色 | 1个 | 2-3个 | >3个 |
   | 业务规则 | 简单 | 有条件分支 | 多层嵌套 |
   | 系统集成 | 无 | 单系统 | 多系统 |
   | 数据处理 | 基础CRUD | 有计算 | 复杂算法 |

   **评分映射**：
   - 1-3分 → 简单 → 快速确认后进入原型
   - 4-7分 → 中等 → 问答澄清后进入原型
   - 8-10分 → 复杂 → 分步确认后进入原型

3. **澄清问答**

   针对遗漏或模糊的信息，主动提问澄清。

### 1.3 需求理解报告模板

```markdown
# 需求理解报告

## 需求摘要
[一句话概括需求核心]

## 用户角色
| 角色 | 描述 | 使用场景 |
|------|------|----------|
| [角色名] | [描述] | [场景] |

## 功能点清单
1. **[功能A]** - [简述]
2. **[功能B]** - [简述]

## 待澄清问题
- Q1: [问题]
- Q2: [问题]

## 复杂度评级
- **级别**：简单/中等/复杂
- **评分**：X/10
- **依据**：[评估依据]
```

### 1.4 澄清提问格式

```markdown
📋 **需求确认**

我理解的需求如下：

**涉及角色**：[角色列表]

**功能模块**：
1. [功能A]
2. [功能B]
...

⚠️ **待澄清问题**：

**Q1: [问题]**
- A. [选项A]
- B. [选项B]
- C. [选项C]
- D. 其他（请描述）

**Q2: [问题]**
[开放性问题]
```

## 阶段2: 原型设计与迭代

### 2.1 核心目标

**先出原型，让用户确认后再生成文档**

设计理念：
- 用户看到原型才能真正确认需求
- 原型迭代是需求澄清的核心环节
- 确认后的原型是文档生成的基础

### 2.2 处理步骤

1. **页面拆解**
   - 识别主要页面
   - 确定页面关系
   - 规划交互流程

2. **HTML 原型生成**

   > **调用 frontend-design skill**：使用 `/frontend-design` 生成高质量原型界面
   >
   > frontend-design 提供独特的美学设计方向，避免通用的 AI 设计风格：
   > - 独特的字体选择（避免 Arial、Inter 等通用字体）
   > - 有记忆点的视觉风格（大胆配色、创意布局）
   > - 精致的动画效果和微交互
   > - 与产品定位匹配的美学调性

   根据需求调性选择设计风格：
   | 产品类型 | 推荐美学方向 |
   |----------|-------------|
   | 企业系统 | 精简/工业风 |
   | 消费应用 | 活泼/有机风 |
   | 金融工具 | 专业/奢华风 |
   | 创意平台 | 极简/杂志风 |

   - 生成可交互的 HTML 原型
   - 支持用户实际操作体验

3. **用户确认与迭代**
   - 用户体验原型
   - 用户反馈修改意见
   - 迭代修改原型
   - 循环直到用户确认

### 2.3 原型输出

| 类型 | 文件 | 说明 |
|------|------|------|
| HTML 原型 | `prototype/index.html` | 可交互体验（核心交付物） |
| 业务流程图 | Excalidraw .md | 业务逻辑流程（辅助理解） |

> **注意**：页面线框图延迟到阶段4生成，阶段2专注于快速产出原型供用户确认，避免因生成线框图耗时过长影响迭代效率。

### 2.4 HTML 原型设计规范

#### 设计理念

| 理念 | 说明 |
|------|------|
| 现代化 UI 元素 | 卡片、阴影、圆角、过渡动画 |
| 开源图标库 | Lucide Icons |
| 工作空间理念 | 突出个性化、高效、舒适 |
| 浅色调为主 | 白色、浅灰主基调 |

#### 视觉规范

```css
:root {
  /* 主色调 */
  --primary-color: #409EFF;
  --success-color: #67C23A;
  --warning-color: #E6A23C;
  --danger-color: #F56C6C;

  /* 文字色 */
  --text-primary: #303133;
  --text-regular: #606266;
  --text-secondary: #909399;

  /* 边框与背景 */
  --border-color: #DCDFE6;
  --bg-color: #F2F6FC;
  --card-bg: #FFFFFF;

  /* 阴影 */
  --card-shadow: 0 2px 12px rgba(0, 0, 0, 0.1);

  /* 圆角 */
  --border-radius-small: 4px;
  --border-radius-base: 8px;
}
```

#### 原型文件结构

```
prototype/
├── index.html          # 入口页面
├── css/
│   └── style.css       # Element UI 风格样式
├── js/
│   └── interactions.js # 交互脚本
└── pages/
    ├── list.html       # 列表页模板
    ├── form.html       # 表单页模板
    └── detail.html     # 详情页模板
```

### 2.5 原型展示与迭代交互

**首次原型展示**：

```markdown
🖥️ **原型已生成**

请体验原型：[原型链接]

## 原型预览
[插入原型截图或描述]

## 主要页面
- [页面A]：[说明]
- [页面B]：[说明]

---

请确认原型是否符合预期：
- ✅ 确认无误，继续生成需求规范
- ✏️ 需要修改（请说明修改内容）
```

**迭代修改交互**：

```markdown
🔄 **原型迭代**

根据您的反馈，已修改原型：

**修改内容**：
- [修改项1]
- [修改项2]

请重新体验原型：[原型链接]

---

请确认：
- ✅ 已满足需求，继续生成需求规范
- ✏️ 还需修改（请说明）
```

### 2.6 输出路径

```
002 领域/产品设计/原型/[项目名称]/prototype/
002 领域/产品设计/原型/[项目名称]/业务流程图.md
```

> **说明**：页面线框图在阶段4统一生成，避免在原型迭代阶段因生成图表耗时过长影响用户确认效率。

### 2.7 阶段完成标志

```markdown
✅ **阶段2完成：原型确认**

用户已确认原型设计，准备生成正式需求规范。

**确认记录**：
- 原型链接：[链接]
- 确认时间：[时间]
- 迭代次数：[次数]

继续生成需求规范文档？
- ✅ 继续
```

## 阶段3: 需求规范生成

### 3.1 核心目标

**生成正式需求规范文档，用于用户签字确认**

用途：
- 正式存档文档
- 用户签字确认依据
- 需求冻结基准

### 3.2 处理步骤

1. **基于确认原型生成文档**
   - 从原型提取界面说明
   - 补充业务规则、数据规则
   - 填充标准模板

2. **文档组装**
   - 按正式模板填充
   - 包含审批记录、版本记录
   - 格式规范化

### 3.3 输出文件

```
002 领域/产品设计/PRD/[项目名称]-需求规范说明书.md
```

### 3.4 文档模板

使用模板文件：[templates/prd-template.md](templates/prd-template.md)

### 3.5 文档审阅交互

```markdown
📄 **需求规范文档已生成**

[[文件路径]]

## 文档摘要
- 功能模块：X 个
- 用户角色：X 个
- 需求条目：X 条

---

请审阅文档内容，确认无误后用于签字：
- ✅ 确认无误，可用于签字
- ✏️ 需要修改（请说明修改内容）
```

### 3.6 阶段完成标志

```markdown
✅ **阶段3完成：需求规范生成**

需求规范文档已生成，可用于用户签字确认。

**输出文件**：
- 📄 需求规范：[[需求规范说明书.md]]

继续生成开发交付的需求传递单？
- ✅ 继续
```

## 阶段4: 需求传递单生成

### 4.1 核心目标

**生成开发交付用的需求传递单**

用途：
- 高效传递给开发人员
- 聚焦关键信息，5分钟可理解
- 开发工作的直接参考

### 4.2 与需求规范的区别

| 维度 | 需求规范（签字版） | 需求传递单（开发版） |
|------|-------------------|---------------------|
| 用途 | 用户签字存档 | 开发工作参考 |
| 详细程度 | 全面详尽 | 聚焦关键信息 |
| 格式 | 正式文档模板 | 轻量传递模板 |
| 读者 | 用户/管理层 | 开发人员 |
| 原型 | 附录索引 | 直接链接嵌入 |

### 4.3 处理步骤

1. **生成页面线框图（低质量快速模式）**
   - 基于已确认的 HTML 原型生成
   - 采用简化样式，快速产出
   - 用于开发人员快速理解页面结构

2. **提取关键信息**
   - 需求名称、来源、优先级
   - 核心功能要点
   - 界面说明（原型链接）
   - 数据规则、边界说明
   - 验收标准

3. **生成传递单**
   - 使用轻量模板
   - 原型链接直接嵌入
   - 流程图附件

### 4.4 输出内容

| 类型 | 文件 | 说明 |
|------|------|------|
| 需求传递单 | MD 文档 | 核心需求信息 |
| 原型链接 | HTML 原型 | 可交互参考 |
| 业务流程图 | Excalidraw | 业务逻辑流程 |
| 页面线框图 | Excalidraw | 页面结构（低质量快速生成） |

### 4.5 输出路径

```
002 领域/产品设计/需求传递/[项目名称]-需求传递单.md
002 领域/产品设计/原型/[项目名称]/prototype/
002 领域/产品设计/原型/[项目名称]/业务流程图.md
002 领域/产品设计/原型/[项目名称]/页面线框图.md
```

### 4.6 需求传递单模板

使用模板文件：[templates/requirement-handover-template.md](templates/requirement-handover-template.md)

## 错误处理

### 输入模糊

```markdown
⚠️ **需求描述较简略**

请补充以下信息：
- 这是为谁解决的问题？
- 期望达成的目标是什么？

示例：财务人员需要快速生成月度报表，减少手工汇总时间。
```

### 需求矛盾

```markdown
⚠️ **发现潜在冲突**

- 描述A：所有用户可查看全部数据
- 描述B：部门数据需要隔离

请确认哪种理解正确？
```

### 阶段回退

```markdown
↩️ **已回退到阶段 X**

当前状态：[状态描述]

请告知需要修改的部分。
```

### 文件冲突

```markdown
⚠️ **文件已存在**

`需求规格说明书.md` 已存在，请选择：
- A. 覆盖原文件
- B. 创建新版本 (v2)
- C. 取消操作
```

## 状态管理

### 会话状态

```yaml
session:
  id: prd-session-{timestamp}
  current_stage: 1-4
  created_at: YYYY-MM-DD HH:mm

  stage_data:
    stage_1:
      status: pending | in_progress | completed
      input: 原始需求
      output: 需求理解报告
    stage_2:
      status: pending | in_progress | completed
      iteration_count: 0-N
      output: HTML原型 + 业务流程图
    stage_3:
      status: pending | in_progress | completed
      output: 需求规范文档路径
    stage_4:
      status: pending | in_progress | completed
      output: 需求传递单 + 原型链接 + 流程图 + 页面线框图
```

### 支持操作

| 命令 | 说明 |
|------|------|
| 继续 | 从当前阶段继续 |
| 回退到原型 | 回退到阶段2重新迭代原型 |
| 回退到规范 | 回退到阶段3修改需求规范 |
| 导出 | 导出当前所有产出 |
| 重启 | 重新开始 |
