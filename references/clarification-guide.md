# 澄清交互指南

本文档定义了需求澄清的交互模式和提问模板。

---

## 1. 澄清触发条件

### 1.1 必须澄清的场景

| 场景 | 示例 | 澄清内容 |
|------|------|----------|
| 信息不足 | "做一个报表系统" | 报表类型、数据来源、用户角色 |
| 存在歧义 | "用户可以查看数据" | 查看范围、权限控制 |
| 逻辑矛盾 | 描述A与描述B冲突 | 确认正确理解 |
| 技术依赖 | "需要对接系统" | 对接方式、接口规范 |
| 流程不完整 | "提交后处理" | 处理人、处理流程 |

### 1.2 可选澄清的场景

| 场景 | 说明 |
|------|------|
| 优化建议 | 可提供更好的解决方案供选择 |
| 边界确认 | 明确功能边界和限制条件 |
| 优先级确认 | 确认功能优先级排序 |

## 2. 提问模式

### 2.1 选择题模式

适用于：有明确选项的问题

```markdown
**Q{n}: [问题描述]**

- A. [选项A描述]
- B. [选项B描述]
- C. [选项C描述]
- D. 其他（请描述）

💡 建议：[推荐选项及理由]
```

**示例**：

```markdown
**Q1: 用户登录方式**

- A. 用户名+密码
- B. 手机号+验证码
- C. 支持两种方式
- D. 其他（请描述）

💡 建议：推荐C选项，提供灵活性同时保证安全性。
```

### 2.2 开放式提问

适用于：需要详细说明的问题

```markdown
**Q{n}: [问题描述]**

请详细说明：
- [需要补充的点1]
- [需要补充的点2]
```

**示例**：

```markdown
**Q2: 审批流程规则**

请详细说明：
- 审批层级和条件
- 超时处理规则
- 驳回后的处理方式
```

### 2.3 表格式确认

适用于：数据字段、权限配置等结构化内容

```markdown
**Q{n}: [字段/配置确认]**

请确认以下内容是否正确，或填写缺失项：

| 项目 | 当前理解 | 需确认/补充 |
|------|----------|-------------|
| [项目1] | [理解内容] | [ ] |
| [项目2] | [理解内容] | [ ] |
```

**示例**：

```markdown
**Q3: 用户数据字段确认**

请确认以下字段是否需要：

| 字段 | 是否需要 | 备注 |
|------|----------|------|
| 用户名 | [ ] | |
| 手机号 | [ ] | |
| 邮箱 | [ ] | |
| 部门 | [ ] | |
| 职位 | [ ] | |
```

## 3. 澄清流程

### 3.1 标准流程

```
┌─────────────────┐
│ 发现需要澄清    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 整理问题列表    │
│ (控制在3-5个)   │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 展示当前理解    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 逐条提问        │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 用户回复确认    │
└────────┬────────┘
         │
         ▼
┌─────────────────┐
│ 更新需求理解    │
└─────────────────┘
```

### 3.2 提问数量控制

| 复杂度 | 问题数量 | 处理方式 |
|--------|----------|----------|
| 简单 | 0-1个 | 可跳过或快速确认 |
| 中等 | 2-3个 | 集中提问，一次确认 |
| 复杂 | 3-5个 | 分批提问，逐步澄清 |

## 4. 交互模板

### 4.1 需求确认开始

```markdown
📋 **需求确认**

我理解的需求如下，请确认是否准确：

**需求摘要**：[一句话概括]

**涉及角色**：[角色列表]

**主要功能**：
1. [功能1]
2. [功能2]
3. [功能3]

---

⚠️ **以下问题需要您确认**：

[问题列表]

---
请逐条回复，或直接回复"确认"继续。
```

### 4.2 矛盾提示

```markdown
⚠️ **发现潜在冲突**

在您的需求描述中，存在以下不一致：

| 描述A | 描述B |
|-------|-------|
| [内容] | [内容] |

**冲突点**：[说明冲突原因]

请确认正确的理解：
- A. 以描述A为准
- B. 以描述B为准
- C. 两者都需要，但场景不同（请说明）
```

### 4.3 信息不足提示

```markdown
⚠️ **信息不足**

以下关键信息缺失，请补充：

**Q1: [问题]**
[选择项或开放式]

**Q2: [问题]**
[选择项或开放式]

---

💡 **提示**：更详细的信息有助于生成更准确的需求文档和原型。
```

### 4.4 确认通过

```markdown
✅ **需求确认完成**

**最终需求理解**：

| 项目 | 内容 |
|------|------|
| 需求摘要 | [摘要] |
| 用户角色 | [角色列表] |
| 功能点 | [数量]个 |
| 复杂度 | [级别] |

即将进入文档生成阶段...
```

## 5. 常见澄清场景

### 5.1 权限范围澄清

```markdown
**Q: 数据访问权限**

用户可以查看的数据范围是：

- A. 仅自己的数据
- B. 本部门的数据
- C. 按角色配置权限
- D. 全部数据

💡 建议：推荐C选项，通过角色配置实现灵活的权限控制。
```

### 5.2 流程细节澄清

```markdown
**Q: 审批流程细节**

请确认审批流程：

1. 提交申请后，第一级审批人是？
   - A. 直接上级
   - B. 部门负责人
   - C. 指定人员

2. 驳回后如何处理？
   - A. 返回修改后重新提交
   - B. 流程结束
   - C. 退回上一级审批人

3. 超时未审批如何处理？
   - A. 自动通过
   - B. 自动驳回
   - C. 提醒后继续等待
```

### 5.3 数据处理澄清

```markdown
**Q: 数据处理规则**

请确认以下数据处理规则：

| 场景 | 当前理解 | 需确认 |
|------|----------|--------|
| 数据新增 | 填写表单提交 | [ ] |
| 数据修改 | 仅创建人可修改 | [ ] |
| 数据删除 | 逻辑删除（保留记录） | [ ] |
| 数据导出 | 管理员可导出Excel | [ ] |
```

## 6. 用户回复处理

### 6.1 回复解析规则

```
用户回复 → 解析内容 → 更新需求理解

示例：
用户: "Q1选A，Q2选C，Q3需要增加一个'岗位'字段"

解析结果：
- Q1: 用户名+密码登录
- Q2: 支持两种登录方式
- Q3: 增加"岗位"字段

更新：更新需求理解报告中的对应内容
```

### 6.2 部分回复处理

```markdown
您已确认：
- Q1: [用户选择]
- Q2: [用户选择]

以下问题仍需确认：
- Q3: [问题]

请继续回复，或回复"跳过"使用默认值。
```

### 6.3 默认值处理

```markdown
以下问题您选择跳过，将使用默认值：

| 问题 | 默认值 |
|------|--------|
| [问题] | [默认值] |

是否同意？回复"确认"继续，或提供您的偏好。
```

## 7. 澄清完成检查

澄清完成后，确保以下内容已明确：

- [ ] 用户角色和权限范围
- [ ] 主要功能点清单
- [ ] 关键业务流程
- [ ] 数据字段和校验规则
- [ ] 特殊需求和限制条件

所有内容明确后，进入文档生成阶段。
