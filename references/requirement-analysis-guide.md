## 2. 用户角色识别

### 2.1 角色识别关键词

| 关键词模式 | 推断角色 |
|------------|----------|
| "用户"、"操作员"、"员工" | 普通用户 |
| "管理员"、"管理"、"后台" | 系统管理员 |
| "领导"、"经理"、"审批" | 管理者 |
| "客户"、"访客" | 外部用户 |
| "财务"、"人事"、"技术" | 业务角色 |

### 2.2 角色提取示例

**输入**：
> 系统管理员可以添加用户，普通用户只能查看自己的数据，部门经理可以审批请假申请

**提取结果**：
| 角色 | 描述 | 权限范围 |
|------|------|----------|
| 系统管理员 | 系统维护人员 | 用户管理、系统配置 |
| 普通用户 | 一般使用者 | 查看个人数据 |
| 部门经理 | 部门负责人 | 审批请假申请 |

## 3. 功能点提取

### 3.1 功能点识别模式

| 模式 | 示例 | 功能点 |
|------|------|--------|
| 动词+名词 | "添加用户" | 用户添加 |
| 可以+动作 | "可以导出报表" | 报表导出 |
| 需要+动作 | "需要发送通知" | 通知发送 |
| 支持+功能 | "支持批量导入" | 批量导入 |
| 能够+动作 | "能够实时监控" | 实时监控 |

### 3.2 功能点分类

| 类别 | 说明 | 示例 |
|------|------|------|
| 数据录入 | 新增、导入、编辑 | 添加用户、批量导入 |
| 数据查询 | 搜索、筛选、列表 | 用户查询、高级搜索 |
| 数据展示 | 详情、报表、图表 | 用户详情、数据统计 |
| 流程操作 | 审批、流转、状态变更 | 请假审批、订单处理 |
| 系统功能 | 登录、权限、配置 | 用户登录、角色管理 |
| 通知交互 | 消息、提醒、导出 | 消息推送、数据导出 |

### 3.3 功能点提取示例

**输入**：
> 需要一个用户管理系统，支持用户注册登录，管理员可以添加删除用户，用户可以修改个人信息，还需要有角色权限管理

**提取结果**：
| 编号 | 功能名称 | 类别 | 优先级 |
|------|----------|------|--------|
| F001 | 用户注册 | 数据录入 | P0 |
| F002 | 用户登录 | 系统功能 | P0 |
| F003 | 用户添加 | 数据录入 | P0 |
| F004 | 用户删除 | 数据录入 | P1 |
| F005 | 个人信息修改 | 数据录入 | P1 |
| F006 | 角色权限管理 | 系统功能 | P1 |

## 4. 歧义点发现

### 4.1 常见歧义类型

| 类型 | 示例 | 问题 |
|------|------|------|
| 范围模糊 | "用户可以查看数据" | 所有数据？还是自己的数据？ |
| 逻辑矛盾 | "所有人可见" + "数据隔离" | 两种要求矛盾 |
| 条件缺失 | "超过限制时提醒" | 什么限制？如何提醒？ |
| 流程不清 | "提交后处理" | 谁处理？如何处理？ |
| 细节遗漏 | "支持导出" | 导出什么格式？ |

### 4.2 歧义识别规则

```
检测规则：
1. 检查"可以"、"能够"后是否有明确对象
2. 检查是否存在相互矛盾的描述
3. 检查条件判断是否有完整的前置条件
4. 检查流程描述是否完整
5. 检查数据操作是否有明确的范围限制
```

### 4.3 歧义发现示例

**输入**：
> 用户可以查看订单，管理员可以管理订单，超时订单需要特殊处理

**发现问题**：
| 歧义 | 问题描述 |
|------|----------|
| 用户查看订单 | 是查看所有订单还是自己的订单？ |
| 管理员管理订单 | 管理包括哪些操作？删除？修改？ |
| 超时订单 | 超时的时间阈值是多少？ |
| 特殊处理 | 具体是什么处理方式？ |

## 5. 复杂度评估

### 5.1 评分维度详解

#### 功能点数量 (权重: 25%)

| 数量 | 得分 |
|------|------|
| ≤3个 | 1分 |
| 4-6个 | 2分 |
| 7-10个 | 3分 |
| 11-15个 | 4分 |
| >15个 | 5分 |

#### 涉及角色 (权重: 20%)

| 角色数 | 得分 |
|--------|------|
| 1个 | 1分 |
| 2个 | 2分 |
| 3个 | 3分 |
| 4-5个 | 4分 |
| >5个 | 5分 |

#### 业务规则复杂度 (权重: 25%)

| 规则类型 | 得分 |
|----------|------|
| 简单增删改查 | 1分 |
| 有基础条件判断 | 2分 |
| 多条件分支逻辑 | 3分 |
| 跨模块业务规则 | 4分 |
| 复杂计算/算法 | 5分 |

#### 系统集成 (权重: 15%)

| 集成情况 | 得分 |
|----------|------|
| 无外部集成 | 0分 |
| 单系统简单接口 | 2分 |
| 多系统集成 | 4分 |
| 复杂数据同步 | 5分 |

#### 数据处理 (权重: 15%)

| 处理类型 | 得分 |
|----------|------|
| 基础增删改查 | 1分 |
| 简单计算统计 | 2分 |
| 复杂报表分析 | 3分 |
| 大数据量处理 | 4分 |
| 实时数据处理 | 5分 |

### 5.2 综合评分公式

```
总分 = 功能点得分 × 25% + 角色得分 × 20% + 业务规则得分 × 25% + 集成得分 × 15% + 数据处理得分 × 15%
```

### 5.3 评分示例

**需求**：
> 建设一个项目管理系统，支持多级计划管理（4级）、资源调配、进度跟踪、风险预警，需要与PDM系统集成

**评分过程**：

| 维度 | 分析 | 得分 |
|------|------|------|
| 功能点 | 4个主要功能模块 | 2分 |
| 角色 | 项目经理、部门主管、管理员、普通员工 | 4分 |
| 业务规则 | 多级计划、资源调配涉及复杂规则 | 4分 |
| 系统集成 | 需要与PDM集成 | 2分 |
| 数据处理 | 进度跟踪、风险预警需要计算 | 3分 |

**计算**：
```
总分 = 2×0.25 + 4×0.20 + 4×0.25 + 2×0.15 + 3×0.15
     = 0.5 + 0.8 + 1.0 + 0.3 + 0.45
     = 3.05
```

**结果**：中等复杂度 (但接近复杂边界，建议按复杂处理)

```markdown

[角色表格，包含：角色名、描述、权限范围、使用场景]

[功能列表，包含：编号、名称、类别、优先级、简述]

## 4. 业务流程
[关键流程的简要描述]

## 5. 待澄清问题
[问题列表，每个问题包含：问题描述、建议选项]

[级别、评分、评估依据]
```

### 6.2 报告生成示例

**输入**：
> 需要一个员工考勤系统，员工可以打卡、查看自己的考勤记录，管理员可以查看所有人的考勤统计、导出报表，需要和门禁系统对接获取打卡数据

**输出报告**：

```markdown
# 需求理解报告

## 1. 需求摘要
为企业管理人员建立员工考勤管理系统，实现考勤数据自动采集、统计分析和报表导出，提升考勤管理效率。

## 2. 用户角色

| 角色 | 描述 | 权限范围 | 使用场景 |
|------|------|----------|----------|
| 普通员工 | 企业员工 | 查看个人考勤 | 日常打卡、查询考勤 |
| 系统管理员 | HR或行政人员 | 全部权限 | 考勤统计、报表导出 |

## 3. 功能点清单

| 编号 | 功能名称 | 类别 | 优先级 | 简述 |
|------|----------|------|--------|------|
| F001 | 员工打卡 | 数据录入 | P0 | 员工进行上下班打卡 |
| F002 | 个人考勤查询 | 数据查询 | P1 | 查看自己的考勤记录 |
| F003 | 考勤统计 | 数据展示 | P0 | 管理员查看全员统计 |
| F004 | 报表导出 | 通知交互 | P1 | 导出考勤报表 |
| F005 | 门禁对接 | 系统功能 | P0 | 获取门禁打卡数据 |

## 4. 业务流程
1. 员工通过门禁打卡 → 数据同步到系统
2. 员工可查看个人考勤记录
3. 管理员查看全员考勤统计
4. 管理员导出考勤报表

## 5. 待澄清问题

**Q1: 打卡方式**
- A. 仅支持门禁打卡
- B. 支持门禁+手机APP打卡
- C. 其他方式

**Q2: 考勤规则**
- 需要了解：上下班时间、迟到早退规则、加班计算规则

**Q3: 报表格式**
- A. Excel格式
- B. PDF格式
- C. 两者都需要

## 6. 复杂度评级

- **级别**：中等
- **评分**：5/10
- **依据**：
  - 功能点：5个（2分）
  - 角色：2个（2分）
  - 业务规则：考勤规则计算（3分）
  - 系统集成：门禁系统对接（2分）
  - 数据处理：统计报表（2分）
```
